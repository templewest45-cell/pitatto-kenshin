# けんしん・できたよ！（仮称） Product Requirements Document

## 1. 概要

### 1.1 目的
特別支援学校の児童生徒が健康診断の流れを事前に練習できる Web アプリを提供し、「見通しが持てない不安」と「感覚過敏による抵抗感」を軽減する。

### 1.2 想定ユーザー
- 利用者: 特別支援学校の児童生徒
- 操作者: 保健室の先生、担任教員

### 1.3 利用環境
- 学校支給タブレット（iPad / Android）
- PC のモダンブラウザ
- Wi-Fi が弱い、または存在しない保健室・検査室

### 1.4 プロダクト方針
- 完全オフラインで動作する PWA とする
- カメラ映像や推論結果は外部サーバーに送信しない
- 児童にとって「次に何が起こるか」が視覚的に分かることを最優先にする
- 教員が児童特性に合わせて難易度調整できることを必須とする

## 2. 成功指標

### 2.1 定性指標
- 児童が検診前に練習内容を理解できる
- 教員が短時間で設定変更できる
- 本番前に不安や拒否反応が軽減したと現場で評価される

### 2.2 定量指標
- 初回インストール後、機内モードで再読み込みしても起動成功率 100%
- タブレットで主要モード実行時の体感フレームレート低下が著しくない
- 教員による 1 セッション開始までの操作ステップ数を 3 ステップ以内に抑える

## 3. スコープ

### 3.1 対象機能
- モード1: じっとする
- モード2: おくちアーン
- モード3: どっちむいてる？
- 教員向け裏設定
- PWA / オフラインキャッシュ
- 児童向けの視覚・聴覚フィードバック

### 3.2 非対象
- 外部サーバー連携
- 個人アカウント管理
- 学校情報システム連携
- 検診履歴のクラウド保存

## 4. ユーザーストーリー

### 4.1 児童
- 児童として、これから何をする練習かを画面で理解したい。そうすれば安心して取り組める。
- 児童として、できている最中にゲージや音で分かりたい。そうすれば続けやすい。
- 児童として、クリアしたら大きく褒められたい。そうすれば達成感を持てる。

### 4.2 教員
- 教員として、児童ごとに目標時間や判定の厳しさを調整したい。そうすれば無理のない練習にできる。
- 教員として、ネットがなくても教室や保健室で確実に使いたい。そうすれば運用が止まらない。
- 教員として、児童がタッチで答えられない場合でも代理入力したい。そうすれば支援方法に幅を持たせられる。

## 5. 機能要件

### 5.1 モード1: じっとする

#### 目的
身体測定、内科検診、心電図、眼科などで必要な「一定時間じっとする」練習を行う。

#### 入出力
- 入力: インカメラ映像、教員設定（検診種別、目標時間、静止閾値）
- 出力: 静止判定、進行ゲージ、成功演出

#### 判定ロジック
- Pose または Face Mesh のキーポイントを取得する
- 鼻、肩など代表点の座標移動量を一定時間窓で計算する
- 移動量が閾値以下であれば静止と判定する
- 静止中のみゲージを進める
- 大きな動きがあった場合はゲージ停止または軽微な減衰を行う

#### UI 要件
- 教員が検診種別を選択できる
- 検診種別に応じたアイコンやイラストを表示する
- カメラ映像はミラー表示する
- 顔出し NG の場合は映像の非表示またはシルエット表示に切り替えられる

### 5.2 モード2: おくちアーン

#### 目的
歯科検診で必要な「口を大きく開け続ける」練習を行う。

#### 入出力
- 入力: インカメラ映像、教員設定（目標時間、開口閾値）
- 出力: 開口判定、進行ゲージ、成功演出

#### 判定ロジック
- Face Mesh を用いて唇周辺ランドマークを取得する
- 上唇中心と下唇中心の距離を計算する
- 距離が閾値以上なら開口中と判定する
- 開口中のみゲージを進める

#### UI 要件
- 口の開け方を示す大きな見本イラストを表示する
- 児童が達成状況を一目で理解できる大きなゲージを表示する
- 音声なしでも理解できる視覚フィードバックを優先する

### 5.3 モード3: どっちむいてる？

#### 目的
視力検査のランドルト環の方向回答を練習する。

#### 入出力
- 入力: タッチ操作、スワイプ操作、教員代理入力
- 出力: 正誤判定、進行ゲージ、成功演出

#### 判定ロジック
- 画面中央にランドルト環を 1 問ずつ表示する
- 4 択ボタンまたはスワイプで方向回答を受け付ける
- 正解時にゲージを進める
- 教員は代理で正解 / 不正解を入力できる

#### UI 要件
- ボタンは大きく、誤タップしにくい配置にする
- 回答方法はタップ優先、スワイプは補助入力として扱う
- 方向表現は上・下・左・右のアイコンでも示す

## 6. 共通 UI / UX 要件

### 6.1 見通し支援
- 進行状況は常に大きなプログレスバーで表示する
- 成功中はゲージが滑らかに伸びる
- 達成時は紙吹雪などの派手なアニメーションを出す
- テキストは短く、ひらがな中心または分かりやすい表現を優先する

### 6.2 音のフィードバック
- 成功判定中は心地よい短い効果音を連続再生できる
- 達成時はファンファーレを再生する
- 聴覚過敏への配慮として一括ミュートできる

### 6.3 教員向け裏設定
- 目立たない導線から設定を開ける
- 想定操作例: 歯車アイコン長押し
- 設定可能項目:
  - 目標時間: 3 / 5 / 10 / 20 秒
  - モード1 の静止閾値
  - モード2 の開口閾値
  - BGM / 効果音 ON/OFF
  - カメラ表示 ON/OFF
  - 検診種別の選択

### 6.4 アクセシビリティ
- 主要操作はタップ 1 回で完了できること
- 配色は高コントラストを確保する
- 音を切っても視覚のみで状態が理解できること

## 7. 非機能要件

### 7.1 オフライン要件
- 初回インストール後は完全オフラインで起動、操作、推論が可能
- Service Worker が HTML / CSS / JS / 画像 / 音声 / AI モデル / wasm をキャッシュする
- AI ライブラリは CDN ではなくプロジェクト内に同梱する

### 7.2 プライバシー要件
- カメラ映像は外部送信しない
- 推論処理はブラウザ内で完結させる
- 個人を識別するデータ保存は行わない

### 7.3 パフォーマンス要件
- 数年前の学校タブレットでも実用速度を維持する
- 必要に応じて入力解像度や推論頻度を落とせる設計にする
- UI アニメーションは推論負荷と競合しないよう軽量にする

### 7.4 信頼性要件
- カメラ権限未許可時に代替説明を出す
- 推論モデルのロード失敗時は再試行導線を出す
- オフライン時でも致命的エラーなしで復旧できること

## 8. 技術要件

### 8.1 基本構成
- HTML5 / JavaScript / CSS による SPA
- PWA 構成
- `navigator.mediaDevices.getUserMedia` でインカメラ取得
- プレビュー映像はミラー表示

### 8.2 推奨技術
- Face Mesh / Pose は MediaPipe を第一候補とする
- 代替として TensorFlow.js を検討可能
- モデル選定基準:
  - ローカル配信可能
  - wasm / モデルファイルを内包できる
  - タブレットで軽量動作する

### 8.3 推奨アーキテクチャ
- `app shell` をプリキャッシュ
- `assets/models` に AI モデル群を配置
- `assets/audio` に効果音を配置
- `settings` は LocalStorage で保持
- 判定ロジックと UI 描画を分離し、各モードを独立モジュール化する

## 9. 受け入れ条件

### 9.1 全体
- ホーム画面から 3 モードへ遷移できる
- 裏設定で変更した値が各モードに反映される
- オフライン再読み込み後もアプリが起動する

### 9.2 モード1
- じっとしている間だけゲージが進行する
- 検診種別変更で表示アイコンが切り替わる
- カメラ非表示設定が反映される

### 9.3 モード2
- 口を一定以上開けた間だけゲージが進行する
- 閾値変更で判定の厳しさが変化する
- オフライン状態でもモデルロードと推論が成功する

### 9.4 モード3
- 4 択またはスワイプで回答できる
- 教員代理入力で正誤を登録できる
- 正解時のみゲージが進行する

## 10. 開発フェーズ

### Phase 1: PoC
- モード2 のみ実装
- PWA 化
- オフラインでのモデルロードと開口判定を検証
- 目標: タブレット実機で「口を開けるとゲージが伸びる」ことを確認する

### Phase 2: 機能拡充
- モード1 とモード3 を追加
- 裏設定メニューを実装
- ミュート、閾値、時間設定を反映

### Phase 3: UI / UX 完成
- 児童向けイラスト差し替え
- エフェクト音と達成演出を組み込み
- 現場テストと調整を実施

## 11. 実装優先順位

1. PWA 基盤とオフラインキャッシュ
2. モード2 の推論パイプライン
3. ゲージと成功演出
4. 裏設定
5. モード1
6. モード3
7. イラスト / 音 / 微調整

## 12. リスクと対策

### 12.1 モデル容量が大きく初回読み込みが重い
- 軽量モデルを優先する
- 初回セットアップ時に必要アセットの事前キャッシュ完了を明示する

### 12.2 低性能端末で推論が重い
- 入力解像度を下げる
- 推論間引き処理を入れる
- モードごとに必要最小限のランドマークだけ利用する

### 12.3 カメラ許可が得られない
- 許可方法を大きく案内する
- モード3 はカメラなしでも利用可能にする

### 12.4 児童によって適切な閾値が大きく異なる
- 裏設定で細かく閾値調整可能にする
- デフォルトはやや易しめに設計する

## 13. 次アクション

### 実装開始時の推奨タスク
1. Phase 1 用の SPA 雛形を作成する
2. ローカル同梱できる Face Mesh 系ライブラリ候補を確定する
3. Service Worker のプリキャッシュ対象一覧を決める
4. モード2 の開口判定閾値設計と UI ワイヤーを作る
5. タブレット実機でオフライン再読み込み試験を行う
